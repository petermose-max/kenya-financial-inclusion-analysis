# Kenya Financial Inclusion Analysis 2024
## FinAccess Household Survey Analysis in SQL and Python 

**Date:** 2024 to 2025  
**Dataset:** FinAccess 2024 Household Survey, FSD Kenya  
**Tools:** PostgreSQL 18, Python 3.13, pandas, SQLAlchemy, Jupyter Notebook

This project analyses the FSD Kenya FinAccess 2024 Household Survey to understand financial inclusion patterns across Kenya's 47 counties. The work demonstrates a professional data analysis workflow using PostgreSQL for database analysis, reusable SQL views, county scorecards, and export-ready outputs for further Python exploratory data analysis. The SQL results show that Kenya has broad financial access driven by mobile money, but formal banking, insurance, financial health, and financial resilience remain uneven across counties and income groups.

## Key Findings

- National mobile money use is high at 78.69%, confirming mobile money as Kenya's strongest financial access channel.
- Bank use is much lower at 45.52%, while insurance use is 19.25%, showing that deeper formal financial inclusion remains more limited.
- Overall access is 81.43% under the working SQL definition, but exclusion is concentrated in counties such as West Pokot, Turkana, Tana River, Narok, and Migori.
- Income inequality is substantial: overall access rises from 68.57% in the lowest income quintile to 92.83% in the highest income quintile.
- Gender gaps are nuanced. The national overall access gap is small, but the national bank-use gender gap is 10.59 percentage points and county-level gaps vary.

## Notebook Guide

| Notebook | Purpose |
|---|---|
| `00_setup.ipynb` | Sets up Python imports, creates the PostgreSQL connection, documents the database structure, and checks core tables and views. |
| `01_sql_basic.ipynb` | Covers SQL Sections 1 to 3: basic exploration, aggregation, grouping, filtering, and cleaning. |
| `02_sql_advanced.ipynb` | Covers SQL Sections 4 to 8: joins, lookup tables, subqueries, CTEs, window functions, views, indexes, materialized views, and final analytical summaries. |
| `03_eda.ipynb` | Planned Python exploratory data analysis using charts, maps, distributions, and survey-weighted summaries. |
| `04_modelling.ipynb` | Planned machine learning notebook for modelling exclusion risk or financial health outcomes. |

## How to Run

1. Install the Python dependencies listed in `requirements.txt`.
2. Ensure PostgreSQL 18 is running locally and that the `finaccess_db` database is available.
3. Create a local `.env` file in the project root using `.env.example` as a template, then set `DB_PASSWORD` to your local PostgreSQL password.
4. Run `notebooks/00_setup.ipynb` to verify the database connection and structure.
5. Run `notebooks/01_sql_basic.ipynb`, then `notebooks/02_sql_advanced.ipynb`.
6. Continue into `03_eda.ipynb` and `04_modelling.ipynb` as the Python analysis phase is developed.

## Data Notes

The raw survey data was split into four PostgreSQL tables because the original file is very wide and contains separate analytical domains. The `finaccess_master` view joins selected variables into a compact respondent-level analysis table, while `finaccess_clean` adds cleaned county codes, county names, and access tier classifications. See `data_dictionary.md` for key column definitions and coding schemes.

The local raw CSV is larger than 100 MB, so it should not be committed to GitHub unless Git LFS is enabled. The `.gitignore` file excludes raw data files by default while keeping the folder structure documented.

The `.env` file is also ignored by Git. This keeps the local PostgreSQL password out of version control while allowing notebooks to connect through `python-dotenv`.
